Idea

With more and more datasets available in the Web that are accessible to a broad range of heterogeneous users the risk of applying inappropriate analysis increases. For example, the Carbon Dioxid (CO2) emissions of coal power plants may be interpolated to arbitrary locations in space by unexperienced users. But how can one interpret the CO2 emissions at an arbitrary location in space where no power plant is located?

We argue that it is only meaningful to predict (or interpolate) values for observable phenomena. This means, in the example of CO2 emissions per power plant, that we cannot observe the CO2 emissions everywhere, but only at locations of power plants. Our aim is to formalise this to make it machine readable and provide basic recommendation for appropriate, or warnings for inappropriate, analysis in statistical software.

Approach

For meaningful spatial statistics, we combine approaches from knowledge engineering that allow to formalize semantics with statistical analysis in order to provide support for a meaningful analysis using appropriate statistical procedures for a particular dataset.

In spatial statistics we distinguish different data types:

Description Example
Spatial Point Pattern Mapping from discrete entities (objects or events) to spatial locations. Location of epicentres of earthquakes
Temporal Point Pattern Mapping from discrete entities (objects or events) to temporal locations. Dates of major floods in a city.
Spatio-temporal Point Pattern Mapping from discrete entities (objects or events) to spatio-temporal locations. Location of epicentres of earthquakes and time of earthquakes.
Marked * Point Pattern Mapping from discrete entities (objects or events) to spatial and/or temporal locations and quality values. Location of epicentres of earthquakes and time of earthquakes with magnitudes.
Geostatistical Variable Mapping from spatial and/or temporal locations to quality values. Landsurface temperature at locations in space and time.
Lattice Variable Mapping from spatial and/or temporal regions to quality values. Number of doctor-prescriptions per consultation in cantons of the Midi-Pyrenees.
Trajectory Mapping from discrete objects and time to spatial locations. Tracks of animals.

The information systems we use to manage and exchange data typically do not make the same distinction. It then becomes very easy to carry out analyses that are syntactically correct but semantically not meaningful, and we will present a number of such cases.

We formalized the notion of meaningfulness in the context of spatio-temporal prediction and spatio-temporal aggregation. Meaningful prediction implies correspondence between the prediction function and the observation function: each prediction should predict an observable quantity. Meaningful aggregation requires correspondence of the observation window, the region or set of points over which data are exhaustive, and the aggregation target (region or set of points). Our theory has been formalized in a higher order logic language (Isabelle), and an OWL pattern is provided for semantic annotation of data sets that could be used to prohibit meaningless spatial statistical operations on these data. Concrete use cases and workflows for the latter will be presented on this website.

Implementation

An overview on the current prototypical implementation is given in the figure below.

There are three main components in the implementation:

  • This Web Portal allows to add new annotations to datasets or browse through existing annotations.
  • The annotations are made available in the LOD cloud using the Parliament triple store, an open source implementation that allows to insert and query Linked Data in the RDF format.
  • There is also an R script available to retrieve the annotations from the Parliament server. Afterwards, an analysis based on our R script for meaningful prediction and aggregation can be performed printing warnings when non-meaningful aggregation or prediction is implemented. The script defines subclasses of the classes for spatial datasets defined in the sp package and added meaningfulness checks based on simple type checks.
  • The annotations are available in the Linked Data Cloud using concepts of our formalism implemented in an OWL.

SPARQL endpoint

The metadata about statistical datasets is stored in the Linked Open Data cloud in the RDF format. The databases in the Linked Open Data cloud are called triple stores and the common query language to retrieve the linked data is called SPARQL. You can retrieve information about the statistical datasets by sending a SPARQL query to the following endpoint of our triple store:

http://giv-mss.uni-muenster.de:8081/parliament/sparql

SPARQL package for R

There is also the R SPARQL package available allowing to directly issue SPARQL requests to a triple store from R and retrieving the results in R.

Theories and Tools

This site presents some theories and tools that you may re-use to implement meaningful statistical analyis. The tools are available as open source in repositories of the Geostatistics Community at the 52Â°North Open Source initiative.

HOL Theory for Meaningful Spatial Prediction and Aggregation

The theory for meaningful spatial prediction and aggregation has been formalised in Higher-Order-Logic (HOL) and implemented and proved in Isabelle/HOL, a semi-automated theorem prover. The theory is available here

OWL Pattern

An ontology pattern written in the OWL is available to to annotate datasets and statistical methods and link them with statistical methods. This allows to implement a recommender tool for statistical methods on datasets, as shown in a prototypical implementation for the R software.

Resources:

R script for Meaningful Spatial Statistics

In order to demonstrate the potential of using the variable types defined in our theory for implementing a recommender tool, we have written an R script, that extends classes from the sp package in R and uses type checks to allow or disallow interpolations or aggregations. The script can be downloaded here.
